## Problem Statement

Boop today is a Claude-Agent-SDK personal agent reachable via iMessage (Sendblue webhook → server → Convex → dispatcher → sub-agents). The iMessage entry point is great for "text it like a person," but Apple's Messages app is a sealed delivery pipe — it cannot stream tokens, cannot run hold-to-talk voice into the agent, cannot show agent status on the lock screen, cannot receive arbitrary content from the system share sheet, and cannot surface in-progress automation runs as Live Activities. A dedicated iOS app that talks to the same boop backend unlocks all of those iOS-native surfaces while preserving iMessage as a fallback. Secondary goal: the build is also the user's path to learning modern iOS (SwiftUI + ActivityKit + WidgetKit + App Intents).

## What Makes This Cool

The "all-three" pitch: voice + share-sheet + widgets, all wired to the same agent and the same memory.

- **Voice-first** — hold-to-talk on the home screen widget or in-app, on-device transcription via Speech.framework, streaming agent response while you're still talking. iMessage cannot do this; Apple's voice memos are sealed audio blobs.
- **Share-sheet superpowers** — share any URL, image, screenshot, or selected text from anywhere in iOS into boop with context. Boop becomes the "do something with this" button for the whole phone.
- **Glanceable agent presence** — Live Activity while an automation runs, lock-screen widget for the latest reply or pending draft, App Intents so "Ask boop X" is a Siri shortcut.

All three are things only a native app can deliver, and all three teach a different slice of modern iOS.

## Constraints

- Solo-developer pace, weekend cadence. Each milestone has to be shippable to TestFlight on its own.
- Must not break the existing iMessage path — the template's "open repo, plug in Sendblue, text your phone" hook stays intact.
- Must reuse the existing dispatcher, memory tiers, automations, draft-and-send flow, and Composio integrations without forking the agent logic.
- iOS distribution is TestFlight / self-install for the maintainer and template forkers. No App Store submission in scope.
- Identity must extend the existing channel model (`sms:+...`, `tg:...`) rather than introducing a parallel user table.

## Premises

These were confirmed before drafting alternatives. If any becomes false, the plan tilts.

1. The iOS app is a new boop channel (`ios:<deviceId>`), parallel to `sms:` and `tg:`. It reuses the dispatcher, memory, automations, and Composio toolkits unchanged.
2. Realtime is split: **inbound** (user → server) is a plain HTTP POST from iOS; **outbound** (server → iOS) is an SSE sidecar on the boop server that streams token-by-token deltas. Convex subscriptions are the source of truth for completed messages (the final assistant message is still written there as today). Sendblue stays as-is for iMessage.
3. Streaming uses an SSE endpoint on the boop server (`GET /channels/ios/stream?conversationId=...`) that subscribes to the existing `broadcast()` pub/sub in [server/broadcast.ts](server/broadcast.ts) filtered by `conversationId`. Per-token broadcasts are added to the Claude Agent SDK streaming callback in [server/interaction-agent.ts](server/interaction-agent.ts). Convex gets the final assistant message only (no per-token mutations).
4. Distribution is TestFlight / self-install. The repo includes the Xcode project; template users open in Xcode, change `BOOP_SERVER_URL` (and Convex deploy URL for backing-store reads) in a config file, hit run.
5. Identity model: boop has **no `users` table** (verified in [convex/schema.ts](convex/schema.ts)). The whole instance is implicitly single-user — `conversations`, `memoryRecords`, `automations` are all unscoped. The `devices` table is therefore an **authorization roster** for this boop instance, not a device→user mapping. Pairing flow: app generates a UUID on first launch, calls `POST /devices/pair/create` which returns a 6-digit code (TTL 10 min, rate-limited to 3 attempts per IP per hour); user pastes the code into the dashboard's Connections tab which marks the device row `paired: true`. From then on, the device's bearer token (issued at pairing time) authorizes inbound HTTP and SSE subscription. Multi-tenancy is handled at deploy level: each template forker runs their own boop, their own devices table.
6. Voice is on-device transcription via Speech.framework (free, private, on-device on iOS 17+). TTS for spoken replies via AVSpeechSynthesizer initially; ElevenLabs upgrade path if voice quality matters.
7. Share extension and Widgets/Live Activities are separate Xcode targets that share an App Group container with the main app for the latest message cache, draft cache, and pairing token.

## Approaches Considered

### Approach A: Narrow start — chat only

SwiftUI app + new `ios:` channel server-side. Send/receive over Convex subscriptions. No voice, no share, no widgets. Ships in ~2 weekends.

- Effort: S
- Risk: Low
- Reuses: dispatcher, memory, Composio, channel abstraction
- Pros: low risk of stalling; validates `ios:` channel before deeper investment
- Cons: doesn't deliver any of the magical moments; voice/share/widgets all become "someday"

### Approach B: Full native, sequenced milestones (RECOMMENDED)

Four milestones, each a TestFlight build, total ~8–12 weekends.

- M1: SwiftUI chat over Convex subscriptions. `ios:` channel server-side, streaming message rows in schema. ~2 weekends.
- M2: Voice — Speech.framework on-device transcription, streaming agent response from server, in-app waveform UI. ~2–3 weekends.
- M3: Share extension — accept URL/image/screenshot/text from anywhere in iOS, post into the same `ios:` channel with a context attachment. ~2 weekends.
- M4: Widgets + Live Activities — home-screen widget for latest reply / pending draft, Live Activity for running automations, App Intent for "Ask boop X" via Siri. ~2–3 weekends.

- Effort: L
- Risk: Medium — scope is broad, each milestone is bounded
- Reuses: everything from A, plus existing automations and draft-and-send system for Live Activity content
- Pros: matches the "all three" scope; each milestone doubles as a TestFlight build and a YouTube clip
- Cons: real commitment; falling off at M2 means voice but no widgets

### Approach C: Companion, no chat app

No chat UI. App Intent for "Ask boop X" (Siri/Shortcuts), Share extension for arbitrary content, Widget showing latest reply/draft, Live Activity for running automations. Replies surface in iMessage and the widget; the "app" is essentially a settings screen for pairing.

- Effort: M (~4–6 weekends)
- Risk: Medium-low
- Reuses: dispatcher, memory, Composio, existing Sendblue path for replies (no new channel needed)
- Pros: genuinely unusual — "boop is in every iOS surface but there's no app to open"; skips chat-list-perf entirely; pure modern iOS frameworks
- Cons: long conversations are painful without a chat surface; you still need iMessage for back-and-forth

## Recommended Approach

**Approach B — Full native, sequenced milestones.**

Rationale: matches the ambitious scope chosen in D2 ("all three"), each milestone is independently shippable so motivation compounds, and the milestones double as YouTube content for the template community. The mixed SwiftUI-first / UIKit-when-forced stack (locked in D4) covers the modern iOS framework surface the user wants to learn.

### M1 — Chat (chat foundation, 2 weekends)

**Architecture decision after eng review:** iOS is **not** registered as a Channel in the `registry` map. The existing [Channel interface](server/channels/types.ts#L32-L49) is webhook-shaped (Sendblue + Telegram inbound webhooks, outbound API calls). iOS reverses that flow: client pushes via HTTP, server pushes back via SSE. Instead, M1 adds a new Express router `server/ios/` that handles inbound POST + SSE outbound directly, calling the existing `runTurn(inbound)` at [server/channels/index.ts:73](server/channels/index.ts#L73). The assistant's final reply still flows through `convex.mutation(api.messages.send, ...)` so dashboard, memory extraction, and consolidation all work unchanged.

**Server-side**
- New router `server/ios/` mounted at `/channels/ios`. Three endpoints:
  - `POST /channels/ios/pair/create` → returns `{deviceId, code, expiresAt}`. Rate-limited to 3 codes per IP per hour. Code TTL 10 min.
  - `POST /channels/ios/pair/consume` (called by dashboard) → marks the device paired, issues a bearer token (stored on the `devices` row as a hashed value, returned cleartext once).
  - `POST /channels/ios/inbound` (auth: bearer token) → parses `(text, attachments?)`, constructs `conversationId = "ios:<deviceId>"`, dedups, calls `runTurn(inbound)` from [server/channels/index.ts](server/channels/index.ts).
  - `GET /channels/ios/stream?conversationId=...` (auth: bearer) → SSE endpoint subscribing to `broadcast()` events filtered by conversationId.
- Wire per-token broadcasts into the Claude Agent SDK streaming callback in [server/interaction-agent.ts](server/interaction-agent.ts) (the SDK exposes streaming via `query()` async iterator — emit `broadcast("assistant_delta", { conversationId, delta, seq })` for each text block). The final `broadcast("assistant_message", ...)` at line 473 stays as-is.
- Add a `devices` table to [convex/schema.ts](convex/schema.ts):
  ```
  devices: defineTable({
    deviceId: v.string(),                  // client-generated UUID
    pairingCodeHash: v.optional(v.string()),  // SHA-256 of 6-digit code, only valid while unpaired
    pairingExpiresAt: v.optional(v.number()),
    paired: v.boolean(),
    bearerTokenHash: v.optional(v.string()),  // SHA-256 of the bearer token, set on pair-consume
    apnsDeviceToken: v.optional(v.string()),  // M4: normal push notifications
    label: v.optional(v.string()),         // user-set ("Lakunle's iPhone")
    pairedAt: v.optional(v.number()),
    lastSeenAt: v.number(),
    createdAt: v.number(),
  }).index("by_deviceId", ["deviceId"]).index("by_pairing", ["pairingCodeHash"])
  ```
- **No `streaming` column on `messages`.** Per the SSE decision (E1), streaming deltas go over SSE, not Convex. The final assistant message is a single `messages.send` mutation when the reply completes, matching the existing Sendblue/Telegram pattern.

**iOS-side**
- New Xcode project: SwiftUI app, deployment target iOS 17. Single window, three tabs (chat / connections / settings).
- Chat list: UICollectionView with diffable data source wrapped in `UIViewControllerRepresentable`. Message bubbles in SwiftUI.
- **Realtime client (no Convex iOS SDK required):** open a long-lived `URLSession` data task to `GET /channels/ios/stream`. Parse SSE events: `assistant_delta` updates the in-progress bubble, `assistant_message` finalizes it, `thinking` shows the "..." indicator. Reconnect with exponential backoff on disconnect.
- **History load (cold start):** one-shot HTTP `GET /channels/ios/messages?conversationId=...&before=...&limit=50` paginated. This is a thin wrapper over a new Convex query — no Convex Swift client needed; the boop server already has a Convex client in [server/convex-client.ts](server/convex-client.ts) it can proxy through.
- First-launch flow: app generates a device UUID (stored in Keychain), calls `POST /channels/ios/pair/create`, displays the 6-digit code. User pastes into dashboard's Connections tab. App polls `POST /channels/ios/pair/check` every 2 seconds (with the deviceId only) until the response includes the bearer token, then stores the bearer token in Keychain and advances to chat.
- Send path: `POST /channels/ios/inbound` with `Authorization: Bearer <token>` header and `(text, attachments?)`.

**Definition of done for M1**
- Type a message on iPhone, see the agent's full reply appear over the SSE stream.
- Live-streaming token UI: assistant bubble visibly grows as `assistant_delta` events arrive (target: first token ≤ 1.5s after send, full reply rendered as fast as the SDK emits).
- iMessage path is unchanged — sending an SMS to the existing Sendblue number still works.
- Pairing code expires after 10 min and cannot be reused. Rate limit (3 codes per IP per hour) enforced.
- Bearer token revocation works: deleting a `devices` row immediately fails subsequent inbound POSTs with 401 and closes any open SSE.

### M2 — Voice (voice in, voice out, 2–3 weekends)

**iOS-side**
- Hold-to-talk button in the chat composer. While held, start `SFSpeechRecognizer` on-device with a live `SFSpeechAudioBufferRecognitionRequest`. Display the partial transcription as the user speaks.
- On release, send the final transcription via the same `POST /channels/ios/inbound` used in M1.
- Reply playback: when an assistant message arrives, expose a "play" button that runs the text through `AVSpeechSynthesizer`. If `ELEVENLABS_API_KEY` is set in the app config, fetch streamed audio from ElevenLabs instead.
- Optional: auto-play assistant replies when the screen is locked or AirPods are connected.

**Server-side**
- No required changes; M1's streaming row is sufficient. Optional: a flag on the inbound action `(source: "voice")` so the agent knows the message came from voice, can adjust verbosity.

**Definition of done for M2**
- Hold the button, speak, release: agent replies in text + optional speech.
- Latency from speech-end to first token under 2 seconds (Speech.framework on-device + streaming agent).

### M3 — Share extension (do-something-with-this, 2 weekends)

**iOS-side**
- New Xcode target: `BoopShareExtension`. Activation rules: URL, image, plain text, attached file under 10 MB.
- UI: a single-screen sheet showing a preview of the shared content + a text field for "what should I do with this?" + Send button.
- On send: upload the attachment to a presigned URL (Convex file storage), then `POST /channels/ios/inbound` with `(deviceId, text, attachmentId)`.
- Share App Group container with main app so the share extension can read the pairing token and last-used hints.

**Server-side**
- Reuse the existing `messages.attachments[]` field added in the recent inbound-attachments PR (commit 7cb4df4).
- The dispatcher already handles attachments — no new code there.

**Definition of done for M3**
- Share a URL from Safari → opens a sheet → "summarize this" → 30 seconds later the summary is in the chat tab.
- Share a screenshot from Photos → "what's this error?" → agent reads the image (existing vision pipeline) and responds.

### M4 — Widgets + Live Activities + App Intents (glanceable agent, 2–3 weekends)

**iOS-side**
- New Xcode target: `BoopWidgets`. Three widget kinds:
  - Small: latest assistant reply (1 line).
  - Medium: latest assistant reply + pending draft (2 lines + action).
  - Lock-screen circular: count of pending drafts.
- Widget timeline updates via Background Tasks + App Group read of the latest cached message.
- Live Activity: starts when an `automationRun` row goes to `running` state for this user; shows automation name + elapsed time + cancel button. Updates via ActivityKit push tokens sent from the server to APNs.
- App Intent: `AskBoopIntent` — `entity: text` parameter, returns the assistant reply. Available in Shortcuts + Siri ("Hey Siri, ask boop to summarize my email").

**Server-side**
- New Convex action `pushLiveActivity({automationRunId, state})` that calls APNs with the activity push token (stored on the `devices` row, sent up at app launch).
- Wire automation execution: when an automation transitions to `running`/`finished`, fire `pushLiveActivity`.

**Definition of done for M4**
- Home-screen widget shows the last reply, refreshes within 1 minute.
- Live Activity appears on the lock screen when an automation starts, disappears when it finishes.
- "Hey Siri, ask boop what's on my calendar" runs the agent and reads the reply.

## Open Questions

- **TTS quality budget.** AVSpeechSynthesizer is free but robotic. ElevenLabs costs real money per character. Pick a default and document the upgrade path in M2.
- **APNs auth key onboarding.** M4 requires an Apple Developer account + APNs auth key (.p8) + Team ID + Key ID configured server-side. Document in the README at M4. APNs Live Activity push tokens are distinct from regular device tokens and expire ~12h after the activity ends — server must handle 410 Gone responses and stop pushing to expired tokens.
- **Reverse proxy / tunnel for SSE in dev.** ngrok free tier closes long-lived HTTP connections after ~2 hours. Document that M1 spike testing needs either ngrok pro (no idle timeout) or Cloudflare Tunnel (no timeout, free).

**Resolved by eng review:**
- ~~Identity model~~ → pairing code with bearer token, rate-limited, no `users` table needed (premise 5 rewritten).
- ~~Convex streaming latency~~ → SSE sidecar chosen (E1, option D). Convex stays source of truth for finalized messages.
- ~~Multi-user template story~~ → handled at deploy level (each fork = own boop = own devices table). No cross-contamination possible because there's no shared backend.

## NOT in Scope

- **App Store submission** — TestFlight / self-install only. Apple review process adds weeks and constrains the template story.
- **Android / cross-platform** — explicitly declined in D4 (Cross-platform tradeoff: defeats the iOS-learning goal).
- **Multi-user accounts within one boop instance** — boop has no `users` table; the deploy model is one-user-per-boop. The `devices` table is a device authorization roster, not a user mapping. Anyone wanting a separate identity forks the repo.
- **Sign in with Apple** — deferred. Pairing code is sufficient for the personal-use + template forker audience. SiwA is a ~30-min swap if pairing feels janky after M1 dogfood.
- **End-to-end encryption** — messages travel in plaintext over TLS to your own boop server. The threat model is "personal agent on your own infrastructure"; E2EE adds key-management complexity that hurts the template story.
- **Background voice (wake word, always-listening)** — explicit hold-to-talk only. Apple's wake-word requires private entitlements and burns battery.
- **iPad / macOS Catalyst** — iPhone only for now. The chat list / share extension / widgets layouts are iPhone-portrait first.
- **Live Activity dynamic island animations** — basic Live Activity in M4. Dynamic Island custom layouts deferred unless cheap to add.

For each new codepath, the realistic production failure mode and whether the plan covers it.

| # | Failure | Codepath | Test? | Error handling? | User sees? |
|---|---------|----------|-------|------------------|------------|
| F1 | Pairing code brute force (1M combos in 6 digits) | `POST /pair/create`, `POST /pair/consume` | M1 unit (rate limit, expiry) | Rate limit + 10-min TTL + code rotation on each create | Generic "code expired or invalid" |
| F2 | Bearer token leaks (e.g. logging accident) | `Authorization: Bearer` parsing | M1 unit (hash-on-store) | Tokens hashed in DB; revoke via row delete; rotate on demand | Re-pair flow |
| F3 | SSE disconnects mid-reply (mobile network) | iOS `URLSession` data task | iOS XCTest (reconnect logic) | Reconnect with backoff; on reconnect, fetch any messages since last seen `seq` | Brief spinner, no message loss |
| F4 | Per-token broadcast not in dispatcher | `broadcast("assistant_delta", ...)` | M1 integration (token deltas observed) | If broadcast fails silently, final message still writes to Convex — degrades to non-streaming | Reply appears whole instead of streaming |
| F5 | Convex `messages.send` write fails after broadcast finishes | `runTurn` exception path | M1 integration (assert message in DB) | Retry once, then log; SSE already streamed the reply | Reply visible in chat but missing from history on cold reload — **CRITICAL GAP** without retry |
| F6 | Speech.framework permission denied (M2) | First-time mic prompt | M2 unit (permission flow) | Show "enable microphone in Settings" link | Clear prompt |
| F7 | APNs Live Activity token expires (M4, 12h after activity ends) | Push delivery | M4 integration (mock APNs 410) | On 410 Gone, mark token expired in `devices` row, stop pushing | Live Activity stops updating; no user-visible error |
| F8 | Share extension memory limit (~120MB on iOS) for large PDFs (M3) | Share extension upload | M3 unit (10MB cap enforced) | Pre-upload size check; reject with toast | "File too large" message |
| F9 | Concurrent inbound from iOS while iMessage turn running | Two-channel turn collision | M1 integration | Existing dedup + turn ID handles it (separate conversationIds) | No-op |
| F10 | `messages.send` failure leaves user-message orphan (no assistant reply) | `runTurn` early throw | M1 unit + integration | Wrap in try/finally; surface error via SSE `error` event | Toast: "Reply failed — tap to retry" |

## Worktree Parallelization

Sequential implementation across milestones (M1 → M2 → M3 → M4) — each milestone's iOS work depends on the prior milestone's server endpoints. Within a single milestone, server-side and iOS-side can run in parallel worktrees:

**Within M1 (after the spike validates SSE):**
- Lane A (server): `convex/schema.ts` (`devices` table), `server/ios/` router, broadcast wiring in `interaction-agent.ts`.
- Lane B (iOS): SwiftUI project skeleton, pairing UI mock, SSE client against a stub server.
- Merge gate: when Lane A is on a branch, Lane B switches its `BOOP_SERVER_URL` to that branch's deploy and integration-tests end-to-end.

**Conflict flag:** both lanes touch the same domain mental-model but no shared file. Safe to parallelize.

## What I noticed about how you think

- You jumped straight to "all three" when given an ambitious-scope option. That's the boil-the-lake instinct that gets you finished apps instead of finished demos. Carry it into M2 — voice is where most builders cut corners.
- You picked SwiftUI-first with UIKit-when-forced over the purist SwiftUI option. That's production-realism over neatness. The chat-list-perf cliff is real and you read past my framing to the actual tradeoff.

- You accepted all premises without pushback.
